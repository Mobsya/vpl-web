VPL3 DOCUMENTATION

About VPL

VPL is a block-based visual programming language. You can program behaviours for Thymio by associating conditions/events to actions.

Creating a program

The program is created in the center of the interface, where the programmer combines events that the robot can detect, with the corresponding actions that the robot needs to perform. To create a program line, simply drag one or more events from the left list of events to the left side of a program line. Then associate one or more actions you want to happen in the right side of the line, taking them from the right list on the screen. A program line must always contain at least one event and one action. Two lines cannot have the same triggering event conditions. If two lines with events occuring at the same time produce conflicting actions, for instance one line states that the robot need to move forward and another backwards, and both line are triggered, the result is undefined.

Events are colored in light orange and actions in light blue. To better distinguish them also in blank/white printing, event blocks have in addition the top-right corner colored in orange:

When completed, a program line can be described by the sentence “when the Event occurs, then the Action is executed”.

Events and event conditions

On the left side of a program line an event can be extended by as many conditions as needed. Conditions allow to better specify the event which is triggered. In the case of the line below, the program line means “when the right button is pushed (event), if the bottom color of Thymio is magenta (condition), then the robot starts moving forward-right (action)”. Conditions  are displayed in a slightly smaller size to indicate the difference with the main event.


While “push the button” is a real event, some others icons (LED stated and internal variables) are representing internal states and not real events. In case one of these icons is set first, the program line will display all conditions at the same level, and all conditions will be considered with the same priority. The line below, for instance, means “when we come to the situation where both the bottom color of Thymio is magenta (condition) and the right button is pressed (condition), then the robot starts moving forward-right (action)”.

Note from the given example that switching events and conditions results in very different behaviors. It’s different to detect the pressing of the button, conditional to the color, and both conditions reunited. If the magenta color appears after pressing the button, the first event is not triggered, while the conjunction of both conditions (whichever occurs first) trigger the second event.

All LEDs and internal states icons can only be conditions and never events. All sensor readings and the clock can be used as events.

Warnings and errors

A code is ready to be executed by the robot when the programmer has included all desired functionalities, and when there are no errors anymore. Errors are visualized with a big red sign, and the error message is displayed at the bottom of the interface on a pale red background. Errors appear if, for instance, there is a code line with an event but no corresponding action. In this case the robot does not know what to do. An error also appears if two code lines have identical events, including parameters and conditions. In this case the robot cannot know what to do.

Sometimes there are code lines that could be executed but do not make much sense, and these generate warning messages. Warning messages are displayed in yellow, and the warning message is indicated in the same bottom location as the errors. Warnings do not prevent the code to be executed, but show potential problems. A warning appears, for instance, if no sensors have been selected in a obstacle detection icon. This can generate unexpected behaviors.

Errors have a priority of display over warnings, as they cause the program not to run. Therefore the message displayed at the bottom is the one for the first error, or for the first warning if there is no error.

Executing a program

PLAY

The green PLAY button means the code has no error, a robot is connected and the code in the interface is not yet running on the robot.

When the programmer presses on PLAY, the code is sent to the robot and executed, and the button changes shape to START AGAIN.

START AGAIN

If the programmer presses START AGAIN, the same code is resent and started again on the robot.

As soon as the code is modified by the programmer, the button goes back to the PLAY state, showing that the code running on the robot is not anymore the one visible in the programming interface. The button goes back to the PLAY state also if the STOP button is pressed.
